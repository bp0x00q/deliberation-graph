# Branching strategies

A branch is useful only when it changes a material assumption, optimization target, causal hypothesis, or system boundary.

Good axes:

- simplest viable implementation;
- maximum robustness or fault tolerance;
- minimum operational dependency;
- adaptation of an existing mechanism;
- clean-slate architecture;
- contrarian/no-build option;
- failure-first or abuse-case design;
- a different causal explanation for the same symptoms;
- centralized versus distributed control;
- eager versus lazy computation.

Avoid role-play labels such as “optimist,” “genius,” or “skeptic” unless they correspond to a precise analytical constraint. Stylistic diversity is not branch diversity.

A first pass should state its assumptions, risks, falsifiers, and next tests before seeing competing conclusions. Shared problem facts and raw evidence are allowed; other branches' conclusions are not.
